# ACM 训练管理平台：简历与面试包装材料

## 1. 项目定位

这个项目更适合包装成：

- 面向 ACM / ICPC 团队的训练管理与数据分析平台
- 面向 ACM / ICPC 团队的训练管理与数据分析平台
- 不只是任务发布，而是把 OJ 同步、画像分析、题目推荐、比赛提醒、异常预警串起来的业务系统

如果你投的是 Java 后端岗位，主标题建议写成：

`ACM 训练管理平台 | Spring Boot + JPA + Vue3 + MySQL/H2 + Docker`

如果你投的是全栈岗位，也可以写成：

`算法训练与竞赛管理平台 | Vue3 + TypeScript + Spring Boot`

## 4. 推荐你主讲的项目亮点

面试不要平均发力，主讲这 4 个点最稳：

### 4.1 OJ 数据同步

你可以这样讲：

`项目里最核心的是把训练系统和真实 OJ 数据打通。学生绑定 Codeforces、AtCoder 账号后，系统会同步 rating、比赛历史和通过题记录；AtCoder 因为公开源拿不到完整做题明细，我补了一套 JSON 导入能力，兼容多种字段名和不同导出结构。`

这个点的价值：

- 不是手填数据，数据可信度更高
- 题目画像、排行榜、推荐题单都能建立在真实训练数据上
- 有外部数据接入、解析、落库、幂等替换、异常处理这些后端含量

### 4.2 隐藏分和推荐逻辑

你可以这样讲：

`我没有直接拿公开 rating 当唯一依据，而是额外设计了隐藏分模型，把 CF rating、ATCoder rating 和做题量结合起来，得到更接近当前训练水平的指标。后续题目推荐、分层训练和异常检测都基于隐藏分。`

这个点的价值：

- 体现你不是只会 CRUD，而是在做规则建模
- 容易被追问“为什么要这么设计”，方便展开
- 和推荐题单、做题分段、异常预警都能串起来

### 4.3 异常预警

你可以这样讲：

`系统会根据最近 24 小时做题频率、最近 7 天高跨度做题等规则识别异常训练行为，例如短时间刷题量明显高于隐藏分阶段阈值，或者连续做超出当前段位很多的题，识别后会落库并支持邮件通知教练。`

这个点的价值：

- 业务上有“判断”和“风控”味道
- 工程上有定时任务、规则触发、告警落库、邮件通知
- 面试官会觉得你考虑了业务管理场景，不只是学生做题页面

### 4.4 比赛提醒和教练端管理

你可以这样讲：

`教练端不仅能管理学生和队伍，还可以同步 Codeforces / AtCoder 官方 upcoming 比赛，补录 QOJ 训练赛，系统按提醒时间推送邮件，减少错过比赛和训练赛安排混乱的问题。`

这个点的价值：

- 体现系统是完整闭环
- 有外部抓取、定时任务、配置化提醒、教练端后台操作

## 5. 一分钟项目介绍

面试开场直接背这一版就够：

`这是我做的一个 ACM 训练管理平台，主要解决训练队里教练难以统一跟踪学生训练数据的问题。技术上我用 Spring Boot 做后端，Vue3 + TypeScript 做前端，系统分教练端和学生端。学生端可以绑定 Codeforces 和 AtCoder 账号，同步 rating、比赛历史和通过题记录；教练端可以统一管理学生、队伍、任务和比赛提醒。`

`我在项目里重点做了三类能力：第一是 OJ 数据同步和 AtCoder 提交导入；第二是隐藏分、积分结算和题目推荐；第三是异常预警和邮件提醒。整体上我希望它不是简单的信息录入系统，而是一个能反映真实训练状态的训练分析平台。`

## 6. 面试官常问问题和回答思路

### 6.1 你为什么要做这个项目？

回答思路：

`普通任务管理系统无法反映算法训练的真实水平，我想把训练任务、真实 OJ 数据、比赛记录和教练管理放到一个系统里，解决训练过程可视化和数据化的问题。`

### 6.2 你做的最有技术含量的模块是什么？

优先回答：

`OJ 同步和隐藏分模型。`

展开：

- OJ 同步涉及外部接口调用、HTML/JSON 解析、字段兼容、异常处理和数据替换
- 隐藏分模型要把多个指标整合成一个可用于推荐和预警的核心指标

### 6.3 隐藏分为什么不直接等于 CF rating？

回答思路：

`因为训练状态不只由公开 rating 决定。有人比赛少但平时做题量大，也有人 rating 高但最近训练不足，所以我把 CF、ATCoder 和做题量结合，做成更适合训练场景的内部指标。`

### 6.4 题目推荐怎么做的？

回答思路：

`先计算隐藏分，再围绕隐藏分生成不同层级的目标难度，比如热身题、核心题、挑战题，然后从题库里选取未做且与目标难度最接近的题。这样推荐不会过于随机，也不会全部集中在一个难度段。`

### 6.5 AtCoder 为什么要支持 JSON 导入？

回答思路：

`因为只依赖官方公开页面时，AtCoder 的比赛历史和 rating 比较容易拿到，但完整做题明细不稳定，所以我设计了导入登录态导出的 JSON 作为补充方案，保证系统里能拿到更完整的训练记录。`

### 6.6 异常预警规则是怎么设计的？

回答思路：

`我先从业务上定义“异常训练行为”，再把它量化成规则，比如 24 小时做题频率明显高于当前隐藏分区间阈值，或者 7 天内连续通过多道远高于隐藏分的题。命中后会落告警表，并支持邮件推送给教练。`

### 6.7 你做了哪些工程化工作？

回答思路：

- 做了前后端分离
- 后端用了缓存减少重复查询
- 提供 Dockerfile 和 Docker Compose 做部署
- 做了定时任务同步和邮件通知
- 写了单元测试覆盖部分核心服务逻辑

### 6.8 项目里最难的点是什么？

回答建议：

`最难的是把“训练水平”抽象成系统里的可计算指标。因为项目既要有真实 OJ 数据，又要把这些数据转成推荐、排行和预警，不是单独做一个同步脚本就结束了，而是要让多个模块围绕一套规则协同工作。`

## 8. 哪些地方可以说，哪些地方不要硬吹

这部分很重要，面试时要诚实。

### 可以说

- 你做了完整的前后端业务闭环
- 你做了外部平台数据接入和清洗
- 你做了规则建模
- 你做了定时任务、邮件通知、Excel 导入、Docker 部署
- 你做了部分测试，且当前后端测试能通过、前端能成功构建

### 不要硬吹

- 不要说自己做了完整生产级安全体系
- 不要说自己做了高并发优化
- 不要说自己实现了复杂推荐算法或机器学习
- 不要说自己用了 JWT/Redis/分布式架构，代码里没有就不要提

### 你需要主动承认的短板

当前项目更接近“可运行的业务系统 / 毕设系统”，不是成熟生产系统，主要短板包括：

- 登录鉴权目前是简化实现，token 规则较轻量，不是正式 JWT 体系
- 密码处理偏演示化，不适合直接用于生产
- 测试覆盖还集中在部分 service，整体覆盖率不高
- 外部 OJ 抓取依赖第三方页面和接口，稳定性受上游影响

如果被问到，你可以这样回答：

`这个项目是我面向训练场景做的完整系统，业务闭环是完整的，但我也知道它距离生产级还有差距，比如鉴权、密码加密、限流、监控和更高覆盖率测试，这些我会作为后续优化方向。`
